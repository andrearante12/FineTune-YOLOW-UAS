# Evaluation Guide

This project includes two evaluation scripts to quantify the trade-off between closed-vocab (VisDrone) and open-vocab detection performance after fine-tuning.

## Prerequisites

```bash
pip install pycocotools
```

Both scripts require:
- Model weight files (base + fine-tuned checkpoints)
- A validation dataset in COCO JSON format
- Images corresponding to the annotation file

## VisDrone Evaluation (Closed-Vocab)

**Script:** `scripts/evaluate.py`

Measures how well each model detects the 10 VisDrone classes (pedestrian, people, bicycle, car, van, truck, tricycle, awning-tricycle, bus, motor). This proves that fine-tuning improved drone-imagery detection.

### Dataset Requirements

VisDrone validation set in COCO format:
```
datasets/VisDrone/
├── images/val/
└── annotations/
    └── val.json          # Generated by yolo_to_coco.py
```

If `val.json` doesn't exist yet, generate it:
```bash
python scripts/yolo_to_coco.py
```

### Usage

```bash
# Auto-discovers all available model checkpoints
python scripts/evaluate.py

# Evaluate a specific fine-tuned model
python scripts/evaluate.py --finetuned-weights 5_epoch.pt

# Custom paths
python scripts/evaluate.py --val-images path/to/images --val-ann path/to/val.json

# CPU-only
python scripts/evaluate.py --device cpu
```

## Open-Vocab Evaluation

**Script:** `scripts/evaluate_open_vocab.py`

Measures how well each model detects open-vocabulary classes (landing pads, backpacks, foreign objects, etc.) that were NOT in the VisDrone training set. This quantifies how much open-vocab capability was retained after fine-tuning.

### Preparing the Dataset

1. **Collect images** — capture screenshots from UAS camera feed that contain your target open-vocab objects

2. **Label in Roboflow** — create a Roboflow project and annotate with bounding boxes. **Important:** name your classes using the same descriptive text you want YOLO-World to use as prompts (e.g., `orange circle landing pad` not `landing_pad`)

3. **Export as COCO JSON** — in Roboflow, export using the "COCO" format

4. **Organize** the exported files:
```
datasets/OpenVocab/
├── images/
│   ├── frame_001.jpg
│   └── ...
└── annotations/
    └── val.json
```

### Usage

```bash
# Auto-reads classes from annotation file
python scripts/evaluate_open_vocab.py

# Custom dataset location
python scripts/evaluate_open_vocab.py --val-images path/to/images --val-ann path/to/val.json

# Override with descriptive text prompts (order must match category_id in annotations)
python scripts/evaluate_open_vocab.py --classes "orange circle landing pad" "backpack" "soccer ball"
```

The `--classes` flag is useful when your Roboflow label names differ from the descriptive prompts you want YOLO-World to use. The script prints the class-to-category mapping so you can verify alignment.

## Interpreting the Trade-Off

Run both evaluations and compare results across models:

| Model | VisDrone mAP | Open-Vocab mAP | Interpretation |
|-------|-------------|----------------|----------------|
| Base (yolov8m-worldv2) | Lower | Higher | Strong open-vocab, weak on drone imagery |
| Fine-tuned (5 epoch, frozen CLIP) | Improved | Similar to base | Best balance — recommended |
| Fine-tuned (40 epoch) | Highest | Degraded | Overfitted to VisDrone classes |

The 5-epoch variant with frozen CLIP encoder should show meaningful VisDrone improvement with minimal open-vocab degradation. If the 40-epoch model scores well on VisDrone but poorly on open-vocab, that confirms the overfitting behavior documented in [Results](./RESULTS.md).

## Metrics Reference

| Metric | Description |
|--------|-------------|
| **mAP@[.5:.95]** | Mean Average Precision averaged over IoU thresholds 0.50 to 0.95 (primary metric) |
| **mAP@.50** | AP at IoU threshold 0.50 (more lenient, counts rough localizations) |
| **mAP@.75** | AP at IoU threshold 0.75 (stricter, requires precise boxes) |
| **mAP Small/Medium/Large** | AP broken down by object size — important for drone imagery where objects are often small |
| **Per-Class AP** | AP for each individual class — shows which classes benefit most from fine-tuning |
| **FPS** | Inference speed — confirms fine-tuning doesn't degrade runtime performance |

## Output

Both scripts save results to `eval_results/`:
- `results.json` — VisDrone evaluation metrics
- `open_vocab_results.json` — Open-vocab evaluation metrics

These JSON files contain the full metrics for all evaluated models and can be used for plotting or generating paper-ready tables.
